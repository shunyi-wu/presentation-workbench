# Presentation Workbench

这是一个用 HTML 制作和管理 PPT 的本地工作台。它不是传统 PPT 模板包，而是一个网页应用：每份 PPT 是一个独立 `deck`，工作台首页负责收纳、预览和进入演示页。

当前系统已经能明显压缩 PPT 制作时间，但还不是成熟产品。复杂生图、画图、长表格、代码截图和部分页面设计仍需要人工与 Codex 继续沟通修正。

## 打开方式

建议使用 Node.js 22 LTS。第一次使用先安装依赖：

```bash
npm install
npx playwright install chromium
```

打开项目文件夹后，VS Code 会读取 `.vscode/tasks.json` 并自动启动工作台。公开版默认地址是：

```text
http://127.0.0.1:5174
```

如果网页被关掉，不需要重新找命令；在 VS Code 里按 `Cmd+Shift+B` 会重新打开工作台桌面页。

也可以在终端手动启动：

```bash
npm run start:workbench
```

## 工作台桌面怎么用

首页是所有 deck 的入口，不是一页普通幻灯片。

- 鼠标滚轮或横向拖动可以浏览 deck 卡片，适合同时管理多个 PPT。
- 点击卡片右下角的封面缩略图，会进入对应 deck 的演示文档页。
- 长按卡片后可以拖动调整顺序，新的顺序会写入 `src/workbenchOrder.json`。
- `placeholder-blank-*` 是占位空白 PPT，只用来展示收纳效果；理解工作台后可以删除。

## 演示页怎么用

进入某个 deck 后，页面使用固定 `1600 x 900` 画布，浏览器只负责整体缩放。

- 左右方向键或底部上一页/下一页按钮可以切换页面。
- 点击 `桌面` 会返回工作台首页。
- 点击 `全屏` 可以进入或退出全屏演示状态。
- 点击 `标注` 或按键盘 `A` 可以打开标注工具，支持画笔、荧光笔、橡皮、撤销和清空当前页。
- URL 中的 `deck` 和 `slide` 参数可以直接定位到某个 deck 的某一页，适合分享具体页面位置。

## 文件夹结构

- `src/`：工作台运行逻辑，例如桌面页、导航、标注工具、deck registry 和基础样式。一般制作具体 PPT 时不需要先改这里。
- `decks/`：每个 PPT 一个独立目录。汇报内容、页面样式、截图评估和交付物优先放在对应 deck 下。
- `scripts/`：辅助脚本，例如启动工作台、清理 review、创建 checkpoint 和检查文档。
- `tests/`：自动检查脚本，用来发现页面溢出、生成截图 review，并验证工作台桌面交互。
- `prompts/`：给 Codex 或其他 AI 使用的任务模板，方便复用常见制作、评估和修复流程。
- `system/`：跨 deck 的长期规则、用户偏好和失败模式，不放某一个具体主题的内容。

## 一个 deck 里面有什么

典型 deck 目录结构：

```text
decks/<deck-id>/
  deck.config.js
  slides.js
  styles.css
  assets/
  review/current/
  deliverables/
```

- `deck.config.js`：注册这个 deck 的 id、标题、默认布局和 renderer。工作台通过它找到并加载这份 PPT。
- `slides.js`：保存页面内容、数据结构和渲染函数。大多数页面制作会集中修改这个文件。
- `styles.css`：保存当前 deck 的页面样式。只影响这份 PPT，不应该随意改全局样式。
- `assets/`：保存当前 deck 使用的图片、截图、生图输出或其他素材。
- `review/current/`：运行 review 后生成截图总览。默认覆盖当前版本，方便反复检查最新结果。
- `deliverables/`：需要把成果对外分享时，再把导出的 HTML、PDF 或其他交付物放到这里。

## 常用检查

修改页面后，通常先检查是否有溢出或越界：

```bash
DECK_ID=<deck-id> npm run check
```

再生成截图 review：

```bash
DECK_ID=<deck-id> npm run review
```

截图输出在：

```text
decks/<deck-id>/review/current/index.html
```

自动检查只能发现一部分问题。遮挡、视觉重心、页面过密、页面过空、叙事是否清楚，仍然需要人工看截图判断。

## 推荐的 AI 协作流程

比较稳定的流程不是一次性让模型生成完整 PPT，而是先把蓝图打磨好，再让 Codex 分批制作：

1. 先确定汇报主题和听众，不急着进入页面制作。
2. 和网页 GPT 交流，得到一份只涉及汇报内容的设计方案报告。
3. 新开聊天窗口评估这份报告，在“评估 -> 修改 -> 再评估”之间循环，直到蓝图足够好。
4. 把最终设计方案交给 VS Code 里的 Codex，让它归档为蓝图并开启计划模式。
5. 先做少量样张确认方向，再继续让 Codex 分批自动制作后续页面。
6. 全部完成后新开窗口写评估方案，确认评估合理后按结果继续修改。

当前系统刚刚够用，不是“一次生成完美 PPT”的系统。复杂生图、画图和部分页面设计仍需要人工跟 Codex 继续交流，但整体已经能大幅压缩制作时间。

## 还在探索的问题

- deck 卡片的制作流程还没有规范：封面预览、卡片文案、排序和归档规则还需要继续沉淀。
- 还没找到足够稳定且高级的视觉语言：不同主题如何统一又不单调仍需要更多样张验证。
- 真实 deck 数量还不够多：不同题材、信息密度和页面类型的压力测试还需要继续积累。
- 系统能力和缺陷还没有完全验证：复杂图表、生图、代码截图、长表格和交互页的边界需要继续摸清。
- 自动化还不完整：目前能检查溢出和导出截图，但审美、叙事节奏和信息取舍仍依赖人工评估。
- 协作流程还可以继续标准化：蓝图、评估报告、修改记录和交付物管理后续可以继续优化。
