# Presentation Workbench / Codex 工作规则

本项目是公开版 HTML 汇报制作工作台。目标是在同一工作区内持续制作不同主题的 deck，同时保持系统层干净、可验证、可归档。

默认使用中文回复和撰写项目内说明，除非用户明确要求其他语言；代码标识、命令、API 名称和专有名词可保留英文。

## 本地预览

- 公开版默认工作台地址固定为 `http://127.0.0.1:5174/`。
- 不要改回 `5173`，避免和私有工作台串端口。
- 不要在每次交付或状态更新里重复提供、链接或主动打开这个网址。
- VS Code 已配置自动启动工作台；用户关闭网页后，可用 `Cmd+Shift+B` 重新打开桌面页。
- 只有用户明确要求打开/排查网址，或验证发现 `5174` 未监听时，才处理浏览器打开、启动服务或端口问题。

## 工作模式

### Planning Mode

用于主题拆解、资料整理、页面蓝图和制作计划。

- 不写页面代码，除非用户明确要求进入制作。
- 先明确汇报目标、时长、听众背景、核心问题和不讲范围。
- 每页先定义表达目标、内容边界和可能的视觉策略。
- 不把单主题讲法、配色、图示选择写入 `system/`。

### Production Mode

用于制作或修改 deck 页面。

- 画布固定为 1600x900，页面内部不得响应式重排，只允许整体缩放。
- 每批优先推进 3-5 页，避免一次性铺满完整 deck。
- 页面设计服务表达目标，不为复杂而复杂。
- 不默认添加总结栏、底部结论条、装饰性模块。
- 交互只在能降低讲解成本时加入。
- 代码、报告、表格、截图等核心材料必须完整可读，不能被装饰挤压。
- 发现越界、遮挡、文本截断或明显视觉失衡时，不得交付。

### Review Mode

用于截图评审、页面修复和质量验收。

- 以截图为准，不只阅读 DOM 或 CSS，也不只看自动测试是否通过。
- 优先检查遮挡、越界、截断、空旷、过密、视觉重心不稳。
- 用户已认可的文案，不因排版修复被随意改写。
- 局部修改只验证受影响页面；全局样式、布局、导航或运行逻辑修改必须扩大验证范围。
- 修改交互、reveal、mode switch、标注层或动态状态时，必须截图验证相关状态，不只验证默认 slide 状态。
- 自动检查通过不等于截图验收通过；页面交付前必须人工查看生成截图或 review 页面。
- 发现明显视觉问题时，即使 `npm run check` 或 `npm run review` 通过，也不得交付。

### Maintenance Mode

用于项目收口、清理、归档、系统规则更新和架构重构。

- 默认做减法。
- 未被入口引用的源码进入删除或归档候选。
- review/current 可覆盖或清空。
- 最终交付物放入当前 deck 的 `deliverables/`。
- 只有跨主题成立的经验才能进入 `system/`。
- 单主题经验放入对应 deck 的 `retrospective.md`。
- 当前目录是公开版 Git 仓库；只有用户明确要求时才提交，不主动添加 remote、不主动 push 到 GitHub。
- 删除历史文件前必须先生成候选清单；大规模清理优先移动到归档目录，确认后再删除。

## Lean System Rules

- 修改前先判断是否真的需要新增文件、组件、layout 或脚本。
- 优先删除、移动、复用、收敛；新增抽象必须说明复用理由。
- 一次性页面结构留在 deck 内，不提升到 `src/`。
- 只有跨 deck 复用成立，才沉淀到 `src/` 或 `system/`。
- 不为规整目录制造空层级。
- 不把单主题经验升级为全局规则。
- 视觉制作允许探索，但系统重构默认做减法。

## 目录职责

- `src/`：通用工作台运行逻辑、桌面页、基础样式和标注工具；不保存具体主题内容。
- `decks/`：每个汇报一个独立目录，保存内容、样式、资产、review、交付物和复盘。
- `scripts/`：启动、review、checkpoint 和校验脚本。
- `tests/`：自动检查和工作台交互测试。
- `prompts/`：给 Codex 或其他编码 AI 使用的任务模板。
- `system/`：跨 deck 的长期规则、用户偏好、失败模式、提示词和模板。

## 复杂视觉与生图

- 制作页面时应主动识别复杂视觉需求；凡是需要承担主要表达任务的封面视觉、概念图、结构图、流程图、状态图、场景化知识图示和复杂插图，都作为生图优先任务处理。
- 复杂视觉优先使用 image2 / imagegen 生成初稿或主体视觉；页面布局、文字排版、基础 UI 和普通信息卡片仍由页面代码承载。
- 涉及精确文字、数字、坐标、箭头关系、表格、品牌标识和严格比例时，应以最终画面是否准确可复核为准；必要时可通过代码绘制、手工标注、截图复刻或局部重绘进行修正。
- 可通过多轮提示词调整、局部重绘、裁切、标注和后处理修正结果；后处理用于修正和整合，不应削弱主体视觉的一致性。
- 修改已有生图前，先查当前 deck `assets/README.md` 中的原 prompt、模型、用途和迭代记录，并查看原始资产。
- 如果只是局部文字、边界、标注、裁切、轻微构图或语义小错，优先用 imagegen edit、局部重绘或后处理修原图，保持原风格和构图；只有主体结构、知识表达或视觉方向整体错误时才重画。
- 新生成或编辑出的版本必须用版本化文件名保存，并把修正 prompt、模型、用途和是否替换正式引用记录回当前 deck 的 `assets/README.md`。
- 如果多轮生成后仍未达到理想效果，不继续无限尝试；应停止迭代，选择当前最佳版本，并向用户反馈已尝试方向、最佳版本的可用点和仍然存在的问题。
- 生图资产必须保存到当前 deck 的 `assets/` 或项目当前输出资产目录，并记录 prompt、模型、用途和必要的后处理说明。
- 未经截图验证的复杂生图、SVG、连线图、状态图、波形图不得直接交付。
- 复杂组件或复杂视觉先在隔离环境、单页或草稿页中开发和截图验证，再接入正式页面。

## 验证规则

按影响范围验证：

- 仅修改蓝图、研究记录、流程说明、提示词、系统文档等不参与页面渲染的文档时，运行 `npm run verify:docs`。
- 修改 `src/`、`index.html`、`vite.config.js`、`playwright.config.js`、`tests/`、设计变量、组件、布局、页面数据或任何影响渲染/构建的文件时，运行 `npm run check` 和 `npm run review`。
- 验证必须针对被修改的 deck；非默认 deck 使用 `DECK_ID=<deck-id> npm run check` 和 `DECK_ID=<deck-id> npm run review`。
- 修改工作台桌面页、registry、启动脚本或测试时，运行相关测试；默认至少运行 `npm run test:gallery`。
- 同时修改文档和页面渲染相关文件时，两类验证都运行。
- 不确定影响范围时，扩大验证范围。

## Review 与交付物

- review 输出应属于具体 deck。
- 默认 review/current 应覆盖，不应无限生成时间戳目录。
- 只有人工确认的版本进入 baseline 或 checkpoints。
- 对外分享文件放入当前 deck 的 `deliverables/`；是否导出单 HTML 由分享场景决定。

## 内置占位

- `placeholder-blank-01` 到 `placeholder-blank-04` 是占位空白 PPT，用来展示工作台收纳多个汇报的能力。
- 不要把这些 deck 描述成正式模版；它们没有实际演示内容。
- 占位空白 PPT 保留网格背景和固定画布即可；用户理解工作台后可以删除。
